# v2.1 Gestures - Design

Drag-to-move, Ctrl+wheel resize, and size/corner persistence for the v2 Rust daemon.
Decided with the owner 2026-07-02: free placement on drag release (no corner snap),
double-click stays inert (fullscreen-guard swallow, unchanged), opacity control rejected
(unverified D3D11 rendering in layered windows - not worth the spike for now).

**Architecture in one line:** all three features are state-file edits applied on the pump
thread; `plan_region` and its six tests stay byte-identical, and the LL mouse hook gains
only lean branches (atomic stores + `PostThreadMessage`, never file I/O or blocking calls).

---

## 1. Verified facts this design stands on

Researched 2026-07-02 (sources fetched, not recalled):

- **Single left click on video is a no-op in stock VLC 3.x** - the win32 vout only acts on
  `WM_LBUTTONDBLCLK` (fullscreen). So drag needs no modifier key and no click swallowing.
  Source: https://raw.githubusercontent.com/videolan/vlc/3.0.x/modules/video_output/win32/events.c
- **Plain wheel = VLC volume; Ctrl+wheel = VLC subtitle scale** (`hotkeys-y-wheel-mode`
  default 0). Windows' "scroll inactive windows" ships ON, so wheel-volume over the PiP
  already works today - document it; the resize gesture MUST swallow Ctrl+wheel.
  Source: https://raw.githubusercontent.com/videolan/vlc/3.0.x/src/libvlc-module.c
- **`MSLLHOOKSTRUCT.mouseData` high word = signed wheel delta** (`WHEEL_DELTA` = 120);
  returning nonzero from `WH_MOUSE_LL` suppresses the event system-wide; reading
  `GetAsyncKeyState(VK_CONTROL)` inside the callback is sound (the staleness caveat applies
  to the key that TRIGGERED the callback, and here that is the wheel).
  Sources: https://learn.microsoft.com/en-us/windows/win32/api/winuser/ns-winuser-msllhookstruct,
  https://learn.microsoft.com/en-us/windows/win32/winmsg/lowlevelmouseproc
- **A swallowed LL button-down never updates async button state**, so "swallow the click,
  then post `WM_NCLBUTTONDOWN`/`SC_MOVE` for a native drag loop" cannot work. Driving the
  move ourselves per mouse-move (the AltSnap model, https://github.com/RamonUnch/AltSnap)
  is the coherent design; `SWP_ASYNCWINDOWPOS` posts cross-queue instead of blocking.
  Source: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-setwindowpos
- **Aspect-locked native edge-resize is impossible externally** (`WM_SIZING` is handled in
  the owner's WndProc only), and our region clips the thick-frame hit zones anyway - so
  wheel resize, not edge resize.
  Source: https://learn.microsoft.com/en-us/windows/win32/winmsg/wm-sizing
- **A `SetWindowRgn` region permanently disqualifies a window from Win11 DWM corner
  rounding** - rounded corners rejected while the minimal-look clip exists.
  Source: https://learn.microsoft.com/en-us/windows/apps/desktop/modernize/ui/apply-rounded-corners
- **Free hotkeys** (verified unbound in VLC core + Qt menus): Ctrl+Alt+plus/minus,
  Ctrl+Alt+Up/Down. Ctrl+Alt+Left/Right is VLC's long jump - never take it.
- **`plan_region` only repositions as part of a size correction** (existing code): a window
  at target size keeps whatever position it has. Free placement therefore survives the
  convergence loop with zero changes to it.

## 2. Corner semantics (the one conceptual change)

`Corner` in state/config becomes **"nearest corner as of the last gesture"**. It is
consulted exactly twice:

1. as the re-anchor target when a resize converges (`plan_region`, unchanged), and
2. as the parking spot on the next PiP enter.

Drag release derives it (without moving the window); resize derives it from the current
position before scaling (so the window re-parks at the corner it is nearest to, not a stale
one). New pure function in `geometry.rs`:

```rust
/// Quadrant of the window center within the work area -> "tl" | "tr" | "bl" | "br".
pub fn nearest_corner(win_left: i32, win_top: i32, win_right: i32, win_bottom: i32,
                      work_left: i32, work_top: i32, work_right: i32, work_bottom: i32) -> &'static str
```

Ties (dead center) resolve to `br` (matches every other fallback in the codebase).

## 3. Drag to move

**Hook side** (`mouse_hook`, all atomics, `Relaxed` - single pump thread):

- New statics: `DRAG_STATE: AtomicU8` (0 idle / 1 armed / 2 dragging), `DRAG_HWND: AtomicIsize`,
  `DRAG_ORIGIN_X/Y`, `DRAG_WIN_X/Y`, `LATEST_X/Y: AtomicI32`, `MOVE_PENDING: AtomicBool`.
- On an **allowed** `WM_LBUTTONDOWN` over the PiP (the existing non-burst branch, right
  where `LAST_ALLOWED_*` is recorded): arm - store cursor origin, `GetWindowRect` origin of
  the cached hwnd, `DRAG_HWND`. Burst-swallowed downs never arm (the fullscreen guard wins;
  a click-then-immediate-redrag inside double-click time therefore does not drag - accepted).
  The down still passes through (VLC ignores it, §1).
- On `WM_MOUSEMOVE`: idle -> one atomic load, fall through (this is the hot path for all
  system mouse movement; keep it to a single branch). Armed and moved beyond
  `SM_CXDRAG`/`SM_CYDRAG` -> dragging. Dragging -> store `LATEST_*`; if
  `!MOVE_PENDING.swap(true)`, `PostThreadMessage(WM_APP_DRAGMOVE)`. Move events are never
  swallowed (VLC sees a harmless hover/drag; gestures module is off by default).
- On `WM_LBUTTONUP`: dragging -> `PostThreadMessage(WM_APP_DRAGEND)`; always reset
  `DRAG_STATE` to idle; the up passes through (pairs with the down VLC already received).

**Pump side** (new `msg.message` arms in `daemon::run`; `WM_APP_*` are thread messages,
`hwnd == 0`, constants `WM_APP`, `WM_APP+1`, `WM_APP+2`):

- `WM_APP_DRAGMOVE`: `MOVE_PENDING.store(false)`; read `LATEST_*`; guard
  `CACHED_HWND == DRAG_HWND != 0`; `SetWindowPos(hwnd, 0, win_origin + (latest - origin),
  SWP_NOSIZE | SWP_NOZORDER | SWP_NOACTIVATE | SWP_ASYNCWINDOWPOS | SWP_NOSENDCHANGING)`.
  Coalescing means at most one queued move at a time; a store racing the pump's read costs
  one redundant `SetWindowPos` (harmless).
- `WM_APP_DRAGEND`: guard `owns_state`; `nearest_corner(current rect, work_area)` ->
  update `state.Corner`, save state, write config (§5); reset the region tracker.
- `maintain_region` gains a 3-line guard: `DRAG_STATE == dragging` -> reset `have_prev`,
  return. (The two-tick debounce already makes mid-drag interference unlikely; the guard
  removes the "user pauses mid-drag for 300ms during unconverged size" edge entirely.)

**Free placement**: nothing repositions on release. The window stays dropped; the next
resize re-parks it at the (freshly derived) nearest corner - per the approved trade-off.

## 4. Ctrl+wheel resize

**Hook side**: `WM_MOUSEWHEEL` over the PiP (`WindowFromPoint` root == cached hwnd) with
`GetAsyncKeyState(VK_CONTROL)` high bit set -> `WHEEL_ACC.fetch_add(delta)` (delta =
`(mouseData >> 16) as u16 as i16`), post `WM_APP_WHEEL` if not pending, **return 1**
(swallow - else VLC scales subtitles). Plain wheel falls through untouched (VLC volume).
Injected events (`LLMHF_INJECTED`) are NOT filtered - `SendInput`-driven smoke checks and
remote-desktop input must keep working.

**Pump side** (`WM_APP_WHEEL`):

- `notches = WHEEL_ACC.swap(0) / 120` (sub-notch remainder discarded - trackpad smooth
  scroll coarsens to notches, accepted).
- Scale current `TargetW` by 5% per notch, applied iteratively `|notches|` times
  (`w = w * 21 / 20` per up-notch, `w = w * 19 / 20` per down-notch), aspect-locked after
  the loop: `new_h = new_w * target_h / target_w`. Integer rounding drift is within
  `plan_region`'s existing +/-2px tolerance.
- Clamp: `TargetW >= 256` and both dimensions `<= 80%` of the current work area (Chrome's
  clamp convention). Clamp violations pin to the bound, aspect preserved.
- Re-derive `Corner = nearest_corner(current rect)`, save state, write config, reset the
  tracker -> existing convergence resizes AND re-parks at that corner with margin.

No new hotkeys in v2.1 (Ctrl+Alt+Up/Down stays reserved; YAGNI until someone asks).

## 5. Persistence - `%APPDATA%\vlc\pip\config.txt`

- **Format**: one line of the existing option tokens, e.g. `w=640 h=360 c=br`. Parsed by
  the existing `parse_options` (whitespace-split), so hand-added `m=`/`min=` tokens work
  for free.
- **Read**: at every enter (daemon and one-shot), effective options =
  `parse_options(config_tokens ++ argv_tokens)` - later duplicates win (pinned by an
  existing test), so argv still overrides. Missing/unreadable config -> defaults + argv.
  Fresh read per enter: the daemon picks up its own gesture writes and hand edits without
  restart. New `options::effective(argv) -> PipOptions` wraps this.
- **Write**: `w={target_w} h={target_h} c={corner}` on drag-end and resize (pump thread,
  never the hook). Write failure swallowed - the gesture still applied for this session
  via the state file.
- **Stated rule** (README): *PiP re-opens at your last size, in the corner nearest to
  where you left it.*

## 7. Accepted edges

- Click, then re-press-and-drag within double-click time/rect: the burst guard swallows
  that down (fullscreen prevention wins), so the drag does not arm. Self-heals after
  `GetDoubleClickTime()`.
- Mixed-DPI monitors: targets are raw pixels; the PiP appears larger/smaller after a
  cross-monitor drag. Per-monitor-V2 awareness already set; no scaling in v2.1.
- VLC dies mid-drag: `WM_APP_*` handlers re-guard on `CACHED_HWND`/`owns_state`; stale
  drags dissolve on the next tick.
- Trackpad smooth-scroll deltas below 120 accumulate in `WHEEL_ACC` until a full notch.

## 8. Rejected (with the reason that killed each)

- **Native modal move loop** (post `WM_NCLBUTTONDOWN`/`HTCAPTION`): community-verified
  only; VLC's Qt frame may handle NC itself; stalls VLC's pump mid-drag; and it cannot be
  combined with swallowing. The AltSnap model wins.
- **Native edge resize** (keep `WS_THICKFRAME`): region clips the hit zones; no external
  aspect lock (`WM_SIZING` is in-process only); fights convergence.
- **Corner snap on release**: owner chose free placement.
- **Double-click action**: owner chose to keep the v2 swallow.
- **Opacity** (`WS_EX_LAYERED` alpha): cross-process API path is fine at same integrity,
  but D3D11 rendering correctness in a layered window is unverified (black-window risk).
  Revisit with a 15-minute spike if wanted.
- **Win11 rounded corners**: mutually exclusive with the `SetWindowRgn` minimal-look clip
  (documented DWM disqualification); GDI round-rect regions are jagged.
- **Lua size-preset submenu**: VLC 3.x supports multi-entry extension menus, but wheel
  resize + persistence covers it with zero Lua risk.
- **Hover overlay controls / multi-monitor throw hotkey / free-position persistence
  beyond nearest-corner**: scope not justified by use.

## 9. Acceptance

- `cargo test`: existing 24 + new units for `nearest_corner` (4 quadrants + center tie),
  wheel scaling (aspect lock, both clamps, negative notches, sub-notch accumulation), and
  config merge precedence (defaults < config < argv).
- `smoke-test.ps1`: existing 21 checks green; new end-to-end checks via `SendInput`:
  Ctrl+wheel over PiP changes status `w/h` aspect-locked; drag moves the window and does
  NOT snap; config file appears with the derived `c=`; enter-after-exit re-opens at
  persisted size/corner; plain wheel is NOT swallowed.
- Manual: drag smoothness (coalesced moves should feel continuous), wheel-volume still
  works, Ctrl+wheel near a work-area edge clamps sanely.

# v2.1 Gestures - Design

Drag-to-move, edge/corner drag-resize, and size/corner persistence for the v2 Rust daemon.
Decided with the owner 2026-07-02: free placement on drag release (no corner snap),
double-click stays inert (fullscreen-guard swallow, unchanged), opacity control rejected
(unverified D3D11 rendering in layered windows - not worth the spike for now). Drags
starting in the outer 16px band of the visible PiP resize (aspect-locked, live,
Firefox-style); center drags move. Ctrl+wheel resize was designed, then dropped by owner
call (drag-resize was the original intent) - **v2.1 intercepts no wheel events at all**.

**Architecture in one line:** gestures act through the pump thread - live coalesced
`SetWindowPos` while a drag is active, state-file + config edits at release; `plan_region`
and its six tests stay byte-identical, and the LL mouse hook gains only lean branches
(atomic stores + `PostThreadMessage`, never file I/O or blocking calls).

---

## 1. Verified facts this design stands on

Researched 2026-07-02 (sources fetched, not recalled):

- **Single left click on video is a no-op in stock VLC 3.x** - the win32 vout only acts on
  `WM_LBUTTONDBLCLK` (fullscreen). So drag needs no modifier key and no click swallowing.
  Source: https://raw.githubusercontent.com/videolan/vlc/3.0.x/modules/video_output/win32/events.c
- **Plain wheel = VLC volume; Ctrl+wheel = VLC subtitle scale** (`hotkeys-y-wheel-mode`
  default 0). Windows' "scroll inactive windows" ships ON, so wheel-volume over the PiP
  already works today - document it. Any wheel gesture of ours would steal a VLC binding,
  which is one reason the wheel stays untouched (§4).
  Source: https://raw.githubusercontent.com/videolan/vlc/3.0.x/src/libvlc-module.c
- **A swallowed LL button-down never updates async button state**, so "swallow the click,
  then post `WM_NCLBUTTONDOWN`/`SC_MOVE` for a native drag loop" cannot work. Driving the
  move ourselves per mouse-move (the AltSnap model, https://github.com/RamonUnch/AltSnap)
  is the coherent design; `SWP_ASYNCWINDOWPOS` posts cross-queue instead of blocking.
  Source: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-setwindowpos
- **Aspect-locked native edge-resize is impossible externally** (`WM_SIZING` is handled in
  the owner's WndProc only), and our region clips the thick-frame hit zones anyway - so
  the self-driven drag-resize of §3, never the native frame.
  Source: https://learn.microsoft.com/en-us/windows/win32/winmsg/wm-sizing
- **A `SetWindowRgn` region permanently disqualifies a window from Win11 DWM corner
  rounding** - rounded corners rejected while the minimal-look clip exists.
  Source: https://learn.microsoft.com/en-us/windows/apps/desktop/modernize/ui/apply-rounded-corners
- **Free hotkeys** (verified unbound in VLC core + Qt menus): Ctrl+Alt+plus/minus,
  Ctrl+Alt+Up/Down. Ctrl+Alt+Left/Right is VLC's long jump - never take it.
- **`plan_region` only repositions as part of a size correction** (existing code): a window
  at target size keeps whatever position it has. Free placement therefore survives the
  convergence loop with zero changes to it.

## 2. Corner semantics (the one conceptual change)

`Corner` in state/config becomes **"nearest corner as of the last gesture"**. It is
consulted exactly twice:

1. as the re-anchor target when a resize converges (`plan_region`, unchanged), and
2. as the parking spot on the next PiP enter.

Drag release derives it (without moving the window); resize derives it from the current
position before scaling (so the window re-parks at the corner it is nearest to, not a stale
one). New pure function in `geometry.rs`:

```rust
/// Quadrant of the window center within the work area -> "tl" | "tr" | "bl" | "br".
pub fn nearest_corner(win_left: i32, win_top: i32, win_right: i32, win_bottom: i32,
                      work_left: i32, work_top: i32, work_right: i32, work_bottom: i32) -> &'static str
```

Ties (dead center) resolve to `br` (matches every other fallback in the codebase).

Drag-resize adds two more pure functions in `geometry.rs`, unit-tested:

```rust
/// Where within the visible rect a drag started: Interior (move) or a resize zone.
/// Band = 16px scaled by the window DPI (caller passes the scaled band width).
/// Corners win over edges when both are within band.
pub enum DragZone { Interior, Left, Right, Top, Bottom, TopLeft, TopRight, BottomLeft, BottomRight }
pub fn classify_zone(pt_x: i32, pt_y: i32, vis: &Rect, band: i32) -> DragZone

/// New window rect for a resize drag: opposite corner anchored (corner zones; dominant
/// relative delta drives the scale) or opposite edge anchored with the perpendicular
/// center fixed (edge zones). Aspect from the start rect. Clamped to
/// [256px wide, 80% of work area], aspect preserved at the bound.
pub fn plan_resize(start: &Rect, zone: DragZone, dx: i32, dy: i32, work: &Rect) -> Rect
```

(`Rect` here is a plain `{left, top, right, bottom}` mirror so `geometry.rs` stays
windows-sys-free, matching `compute_corner`'s convention.)

## 3. Drag gestures - move and resize

**The zone decides the mode.** At arm time the visible rect = window rect intersected with
the region box (`GetWindowRgn` + `GetRgnBox`, cheap GDI; no region -> the window rect).
`classify_zone(pt, vis, band)` with band = 16px x `GetDpiForWindow`/96: `Interior` -> move,
edge/corner zone -> resize.

**Hook side** (`mouse_hook`, all atomics, `Relaxed` - single pump thread):

- New statics: `DRAG_STATE: AtomicU8` (0 idle / 1 armed / 2 moving / 3 resizing),
  `DRAG_ZONE: AtomicU8`, `DRAG_HWND: AtomicIsize`, `DRAG_ORIGIN_X/Y: AtomicI32`,
  `DRAG_START_*` (window rect at arm, 4x AtomicI32), `LATEST_X/Y: AtomicI32`,
  `MOVE_PENDING: AtomicBool`.
- On an **allowed** `WM_LBUTTONDOWN` over the PiP (the existing non-burst branch, right
  where `LAST_ALLOWED_*` is recorded): arm - store cursor origin, window rect, `DRAG_HWND`,
  zone. Burst-swallowed downs never arm (the fullscreen guard wins; a
  click-then-immediate-redrag inside double-click time therefore does not drag - accepted).
  The down still passes through (VLC ignores it, §1).
- On `WM_MOUSEMOVE`: idle -> one atomic load, fall through (this is the hot path for all
  system mouse movement; keep it to a single branch). Armed and moved beyond
  `SM_CXDRAG`/`SM_CYDRAG` -> moving or resizing, per the armed zone. Active -> store
  `LATEST_*`; if `!MOVE_PENDING.swap(true)`, `PostThreadMessage(WM_APP_DRAG)`. Move events
  are never swallowed (VLC sees a harmless hover/drag; gestures module is off by default).
- On `WM_LBUTTONUP`: active -> `PostThreadMessage(WM_APP_DRAGEND)`; always reset
  `DRAG_STATE` to idle; the up passes through (pairs with the down VLC already received).
- Injected events (`LLMHF_INJECTED`) are NOT filtered - `SendInput`-driven smoke checks
  and remote-desktop input must keep working.

**Pump side** (new `msg.message` arms in `daemon::run`; `WM_APP_*` are thread messages,
`hwnd == 0`, constants `WM_APP` and `WM_APP+1`):

- `WM_APP_DRAG`: `MOVE_PENDING.store(false)`; read `LATEST_*`; guard
  `CACHED_HWND == DRAG_HWND != 0`. **Moving**: `SetWindowPos(hwnd, 0, start_pos + (latest -
  origin), SWP_NOSIZE | SWP_NOZORDER | SWP_NOACTIVATE | SWP_ASYNCWINDOWPOS |
  SWP_NOSENDCHANGING)`. **Resizing**: on the first application drop the region
  (`SetWindowRgn(null)` - VLC's mini chrome shows until release re-clips), then apply
  `plan_resize(start_rect, zone, latest - origin, work_area)` with the same flags minus
  `SWP_NOSIZE`. Coalescing means at most one queued message at a time; a store racing the
  pump's read costs one redundant `SetWindowPos` (harmless).
- `WM_APP_DRAGEND`: guard `owns_state`. **Moving**: `nearest_corner(current rect,
  work_area)` -> update `state.Corner`, save state, write config (§5), reset the region
  tracker. **Resizing**: additionally set `TargetW/H` first - the final size minus the
  chrome measured at drag start when `min=1` (no video child at start -> chrome 0), or the
  final window size when `min=0` - then corner/save/config/reset as above. Convergence
  re-clips the region and issues at most a +/-2px correction, so there is no post-release
  jump.
- `maintain_region` gains a 3-line guard: `DRAG_STATE >= moving` -> reset `have_prev`,
  return. (The two-tick debounce already makes mid-drag interference unlikely; the guard
  removes the "user pauses mid-drag for 300ms during unconverged size" edge entirely.)

**Free placement**: nothing repositions on release. A move stays dropped; a resize keeps
its anchor. The freshly derived nearest corner is consumed only by the next enter and by
any future convergence-driven re-park (e.g. VLC chrome changes while parked).

## 4. Wheel - untouched

v2.1 intercepts no wheel events. Plain wheel over the PiP is already VLC volume (§1) and
Ctrl+wheel stays VLC's subtitle scale. A Ctrl+wheel resize was designed, then dropped by
owner call once drag-resize covered the original intent; it remains trivially revivable
(the same `TargetW/H` state edit §3 performs on resize release). No new hotkeys either
(Ctrl+Alt+Up/Down stays reserved; YAGNI until someone asks).

## 5. Persistence - `%APPDATA%\vlc\pip\config.txt`

- **Format**: one line of the existing option tokens, e.g. `w=640 h=360 c=br`. Parsed by
  the existing `parse_options` (whitespace-split), so hand-added `m=`/`min=` tokens work
  for free.
- **Read**: at every enter (daemon and one-shot), effective options =
  `parse_options(config_tokens ++ argv_tokens)` - later duplicates win (pinned by an
  existing test), so argv still overrides. Missing/unreadable config -> defaults + argv.
  Fresh read per enter: the daemon picks up its own gesture writes and hand edits without
  restart. New `options::effective(argv) -> PipOptions` wraps this.
- **Write**: `w={target_w} h={target_h} c={corner}` on drag-end, move and resize alike
  (pump thread, never the hook). Write failure swallowed - the gesture still applied for
  this session via the state file.
- **Stated rule** (README): *PiP re-opens at your last size, in the corner nearest to
  where you left it.*

## 7. Accepted edges

- Click, then re-press-and-drag within double-click time/rect: the burst guard swallows
  that down (fullscreen prevention wins), so the drag does not arm. Self-heals after
  `GetDoubleClickTime()`.
- No sizing cursors over the resize band: `SetCursor` only works from the owning thread's
  WndProc, so cursor feedback would need injection. The band is an invisible affordance
  (Firefox's PiP edges are equally unmarked).
- The outer 16px of the visible PiP is resize territory; moves grab the interior. A very
  small PiP (near the 256px clamp) still leaves an interior wider than 220px.
- During a resize drag the region is dropped, so VLC's mini chrome is visible until
  release re-clips - transient by design.
- Resize while playback is stopped (no video child): chrome approximated as 0; the size
  fully converges on next play.
- Mixed-DPI monitors: targets are raw pixels; the PiP appears larger/smaller after a
  cross-monitor drag. Per-monitor-V2 awareness already set; no scaling in v2.1.
- VLC dies mid-drag: `WM_APP_*` handlers re-guard on `CACHED_HWND`/`owns_state`; stale
  drags dissolve on the next tick.

## 8. Rejected (with the reason that killed each)

- **Native modal move loop** (post `WM_NCLBUTTONDOWN`/`HTCAPTION`): community-verified
  only; VLC's Qt frame may handle NC itself; stalls VLC's pump mid-drag; and it cannot be
  combined with swallowing. The AltSnap model wins.
- **Native edge resize** (keep `WS_THICKFRAME`): region clips the hit zones; no external
  aspect lock (`WM_SIZING` is in-process only); fights convergence. Superseded by the
  self-driven band (§3), which has none of those problems.
- **Ctrl+wheel resize**: designed, then dropped by owner call - drag-resize was the
  original intent and covers it, and dropping means zero wheel interception (VLC keeps
  both its wheel bindings).
- **Corner snap on release**: owner chose free placement.
- **Double-click action**: owner chose to keep the v2 swallow.
- **Opacity** (`WS_EX_LAYERED` alpha): cross-process API path is fine at same integrity,
  but D3D11 rendering correctness in a layered window is unverified (black-window risk).
  Revisit with a 15-minute spike if wanted.
- **Win11 rounded corners**: mutually exclusive with the `SetWindowRgn` minimal-look clip
  (documented DWM disqualification); GDI round-rect regions are jagged.
- **Lua size-preset submenu**: VLC 3.x supports multi-entry extension menus, but wheel
  resize + persistence covers it with zero Lua risk.
- **Hover overlay controls / multi-monitor throw hotkey / free-position persistence
  beyond nearest-corner**: scope not justified by use.

## 9. Acceptance

- `cargo test`: existing 24 + new units for `nearest_corner` (4 quadrants + center tie),
  `classify_zone` (all 9 zones, corner precedence where bands overlap, band scaling),
  `plan_resize` (aspect lock, both clamps, each anchor: 4 corners + 4 edges), and config
  merge precedence (defaults < config < argv).
- `smoke-test.ps1`: existing 21 checks green; new end-to-end checks via `SendInput`:
  interior drag moves the window (same size, no snap); corner drag changes status `w/h`
  aspect-locked; a band click without movement passes through untouched; config file
  appears with the derived `c=`; enter-after-exit re-opens at persisted size/corner;
  plain wheel and Ctrl+wheel are both NOT swallowed.
- Manual: drag smoothness (coalesced moves should feel continuous), resize-band feel at
  the 256px clamp, wheel-volume still works, chrome flash during a resize drag is
  acceptable.
